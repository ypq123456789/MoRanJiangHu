# 小说分解-主剧情章节联动改造计划

## 文档目的

这份文档用于锁定“小说分解 -> 主剧情 -> 规划分析 -> 世界演变 -> 变量状态”整条链路的统一改造方案，避免后续边改边漂、遗漏关键联动点。

本次改造的核心目标不是单独优化某一段提示词，而是建立一套稳定的“原著章节推进系统”：

1. 小说分解负责产出可复用的原著章节资产。
2. 主剧情按“上一章 / 当前章 / 下一章”滑窗注入区间剧情概括。
3. 规划分析在同人模式下负责维护“当前对齐原著第几章”与“是否满足换章条件”。
4. 世界演变与变量层围绕同一章节状态树同步。

## 一、当前问题总结

### 1. 小说分解输出层

- 列表区块当前还混杂“建议使用 `[1] 内容` 格式”这类前端文案，干扰编辑体验。
- `[索引]` 目前部分依赖 AI 输出，不稳定，容易错号、漏号或格式错乱。
- `登场角色` 还没有明确要求将“名字 + 身份”合并成单条结构，导致后续注入时角色识别不稳定。
- `本章节开启条件` 的职责定义不够明确，容易退化成普通摘要，而不是“剧情衔接条件”。
- 分解结果虽然按章节拆分，但缺少统一的“对应章节号 / 对应章节标题”锚点输出语义。

### 2. 主剧情注入层

- 当前主剧情仍是读取整份小说分解快照裁剪文本，不是基于“当前原著章”的滑窗注入。
- `小说拆分主剧情注入上限` 这一配置语义不准确，因为主剧情真正需要的是“上一章 / 当前章 / 下一章”的完整区间概括，而不是模糊的全文裁剪上限。
- 主剧情尚未根据“当前原著章推进状态”自动切换注入章节。

### 3. 规划分析层

- 规划分析虽然能吃小说分解注入，但没有一个稳定的章节推进变量来维护“当前对齐原著第几章”。
- 同人模式下，规划分析还没有强制复用原著章节标题，导致原著章节与游戏内章节两套标题可能漂移。
- 规划分析尚未承担“判断当前原著章是否完成、是否切换到下一章”的职责。

### 4. 剧情状态 / 变量层

- `剧情.当前章节` 当前更偏向游戏内剧情章节，不是“原著章节对齐状态”。
- 缺少专门字段用于保存：
  - 当前原著章节序号
  - 当前原著章节标题
  - 当前原著章节区间摘要
  - 当前原著推进状态
  - 当前原著换章条件
  - 当前原著切换说明

### 5. 联动机制层

- 小说分解、主剧情、规划分析、世界演变目前还没有围绕同一个章节状态做统一推进。
- 时间线虽然能做连续性约束，但“换章”不应只靠时间，应优先根据剧情推进与章节条件判断。

## 二、改造后的目标架构

### 1. 小说分解的职责

小说分解只做“原著章节资产生成”，不直接负责章节切换。

每个分段至少要提供：

- 对应章节号
- 对应章节标题
- 区间剧情概括
- 原著承接锚点
- 区间剧情时间线
- 起始时间
- 结束时间
- 本章节开启条件
- 本章节结束条件
- 登场角色
- 角色关系

其中：

- `本章节开启条件` 定义为：进入本章前必须已经成立、并且本章正文必须承接的剧情状态。
- `登场角色` 定义为：使用 `名字(身份)` 形式输出，例如 `萧炎(萧家少主)`。
- `[索引]` 不由 AI 输出，由前端展示层统一生成。

### 2. 主剧情的职责

主剧情只消费章节滑窗概括，不直接消费整棵小说分解结构树。

固定注入结构：

- `【上一章区间剧情】`
- `【当前章区间剧情】`
- `【下一章区间剧情】`

规则：

- 第一章时没有上一章。
- 最后一章时没有下一章。
- 当前章永远由 `剧情.当前章节.原著章节序号` 决定。
- 当规划分析确认当前章完成后，主剧情下一回合自动切换滑窗。

### 3. 规划分析的职责

规划分析是“原著章节推进调度器”，尤其在同人模式下承担章节对齐判断。

需要负责：

- 判断当前原著章节是否仍在推进中
- 判断当前原著章节是否已完成
- 判断是否满足换章条件
- 完成后把剧情状态切换到下一原著章节
- 同步修订原著章节标题、摘要、换章说明

### 4. 世界演变的职责

世界演变仍然以世界层结果为主，但章节上下文以当前原著章为核心锚点。

### 5. 变量层的职责

变量层负责把章节推进状态固化到 `剧情.*` 结构，供主剧情、规划分析、变量校准和 UI 共用。

## 三、建议新增或扩展的数据结构

## 3.1 `剧情.当前章节` 扩展字段

建议新增以下字段：

- `原著章节序号: number`
- `原著章节标题: string`
- `原著区间摘要: string`
- `原著推进状态: "未开始" | "推进中" | "已完成"`
- `原著换章条件: string[]`
- `原著切换说明: string`

如有必要，可继续补充：

- `原著上一章标题: string`
- `原著下一章标题: string`

但第一阶段不强制新增，避免状态树过胖。

## 3.2 小说分解分段结构扩展

建议在小说分解分段结构中补充或强化以下语义：

- `起始章序号`
- `结束章序号`
- `标题`
- `概括`
- `章节开启条件`
- `章节结束条件`
- `登场角色`

并确保注入组装时可直接生成：

- 当前章标题
- 当前章区间概括
- 当前章换章条件

## 四、提示词改造原则

### 4.1 小说分解提示词

必须改成以下规则：

- 不再要求 AI 写 `[1]`、`[2]` 这类索引前缀。
- 列表区块输出纯列表内容即可。
- `登场角色` 强制写成 `名字(身份)`。
- `本章节开启条件` 明确定义为“剧情衔接条件”，不是泛化摘要。
- 输出中应明确当前处理的是第几章、章节标题是什么。
- 时间线连续性仍保留：若存在上一章节结束时间，则当前章节必须与之衔接。

### 4.2 主剧情相关提示词

要让主剧情明确理解：

- 它现在收到的是原著滑窗概括
- 当前章是核心推进目标
- 上一章只做衔接
- 下一章只做预备锚点

### 4.3 同人模式下的规划分析提示词

必须强化以下判断：

- 当前回合对齐的原著章节是谁
- 当前原著章是否已经被完成
- 当前变动属于延续原著、偏转原著还是旁支展开
- 若当前章已完成，是否切章
- 同步把 `剧情.当前章节.标题` 与原著章节标题对齐

## 六、风险清单

### 1. 旧存档兼容风险

- 旧存档没有新增章节字段。
- 解决方式：在 `storyState` 规范化逻辑中补默认值。

### 2. AI 输出兼容风险

- 旧提示词残留可能仍输出 `[索引]`。
- 解决方式：解析层继续兼容旧索引格式，但新提示词不再要求。

### 3. 规划分析误切章风险

- 如果换章条件定义不清，规划分析可能提前切章。
- 解决方式：提示词中明确“换章必须优先依据当前章完成条件，不只依据时间推移”。

### 4. 注入文本过长风险

- 三章滑窗在极端情况下仍可能很长。
- 解决方式：滑窗结构优先完整，其次再做安全截断，但不能破坏三段结构。

## 七、验收场景

### 场景 1：小说分解输出

- AI 输出中不再带 `[1]`。
- 前端编辑器仍能看到 `[1]`、`[2]`。
- `登场角色` 形如 `张无忌(明教教主候选)`。

### 场景 2：主剧情注入

- 当前对齐原著第 5 章时：
  - 主剧情收到第 4、5、6 章区间剧情概括
  - 而不是整篇小说分解快照

### 场景 3：规划分析切章

- 当前章条件已达成后，规划分析输出命令把原著章节序号切到下一章。
- 下一回合主剧情滑窗自动前移。

### 场景 4：同人模式

- `剧情.当前章节.标题` 与原著章节标题保持一致或默认同步。
- 规划分析明确判断本回合属于：
  - 延续原著
  - 偏转原著
  - 旁支展开
