# 项目功能梳理与重构任务清单

## 1. 当前功能分类

### 1.1 核心游戏循环
- 新开游戏：世界观生成、角色建档、开局初始化。
- 主剧情回合：输入 -> 拼接上下文 -> 调用模型 -> 解析结构化响应 -> 写入状态/记忆/历史。
- 重 Roll / 重解析：对最近回合做回档、原文修复、重新落命令。
- 自动存档 / 手动存档 / 读档。

### 1.2 世界与数值系统
- 角色：属性、血量、金钱、装备、物品、功法、BUFF、突破条件。
- 环境：时间、地点、天气、节日、环境变量。
- 世界：地图、建筑、活跃 NPC、进行中事件、已结算事件、江湖史册。
- 战斗：敌方列表、战斗状态与战斗结束自动清理。
- 门派：玩家门派、职位、贡献、兑换、任务。
- 任务与约定：独立列表展示与命令落地。
- 剧情系统：章节、下一章预告、历史卷宗、近中长期剧情规划、待触发事件、关键剧情变量组、群像镜头排期。
- 女主剧情规划：女主档案、互动排期、镜头规划、推进引导。

### 1.3 提示词与上下文系统
- 默认提示词池：核心、数值、难度、写作、自定义。
- 运行时上下文装配：
  - AI 角色声明
  - 世界观与世界状态
  - NPC 在场/离场档案
  - 记忆系统
  - 剧情/女主剧情规划
  - 输出协议与行动选项
  - 风格、人称、字数、免责声明
- COT 变体：
  - 主 COT
  - 女主规划 COT
  - NTL 女主规划 COT
  - 开局 COT
  - 世界观生成 COT
  - 回忆检索 COT
  - 正文润色 COT
- 酒馆预设：导入、启用、顺序项启停、槽位切换、内置占位符注入。

### 1.4 AI 功能
- 主剧情生成
- 世界观生成
- 开局生成
- 世界演变
- 记忆回忆检索
- 记忆总结
- 正文润色
- NPC 生图词生成
- 图片生成
- 接口连通性测试

### 1.5 UI 功能
- 主聊天界面
- 左/右侧信息面板
- 移动端快捷菜单
- 角色、背包、装备、战斗、社交、队伍、功法、世界、地图、门派、任务、约定、剧情、女主规划、记忆、图册等弹窗
- 设置中心：API、记忆、提示词、世界、游戏、现实模式、酒馆预设、历史查看、上下文查看、视觉、主题、存储

## 2. 重构前主要结构问题

### 2.1 `hooks/useGame.ts`
- 重构前 6223 行，职责混杂。
- 同时承担：
  - 游戏设置规范化
  - 酒馆预设规范化
  - COT/提示词切换与上下文装配
  - 记忆总结
  - NPC 生图队列
  - 世界演变
  - 开局流程
  - 发送流程
  - 自动重试
  - 存档读写协调
- 结果：任何一个功能改动都容易连带破坏主流程。

### 2.2 `services/aiService.ts`
- 重构前 2489 行，协议适配、流式解析、标签修复、结构化解析、图片接口、业务 API 全部混在一起。
- 同时保留 OpenAI / Gemini / Claude / DeepSeek 多套文本调用路径。
- 文本传输层和业务层没有边界，导致后续很难统一为 chat completions。

### 2.3 提示词装配逻辑不稳定
- COT 多版本切换发生在运行时，并通过覆盖 `core_cot` 的 `id/content` 完成。
- 女主规划与 NTL 版本也在运行时开关里“改写提示词池”。
- 酒馆预设注入时存在内置标识符替换与二次改写，缺少统一后处理层。
- 结果：当前实现更像“拼文本 + 重命名注入”，而不是“直接引用不同提示词文件”。

### 2.4 旧兼容逻辑过多
- `启用兼容模式` 仍深度参与主流程。
- `旧版默认额外系统提示词`、`legacy_preset_*`、旧版 API 迁移字段、旧版标签兼容修复等逻辑分散。
- `useGameState` 与 `useGame` 都维护一套游戏设置/酒馆预设规范化，存在重复实现。

## 3. 已确认的重点错误与不合理点

### 3.1 COT 版本切换方式错误
- 现状：把不同 COT 模板内容覆盖写进 `core_cot`，再继续按同一个逻辑注入。
- 目标：运行时直接选择不同来源文件，不再做“模板改名覆盖”。

### 3.2 文本 API 调用未统一
- 现状：文本请求仍存在 Gemini `generateContent`、Claude 原生 messages、OpenAI 兼容 chat completions 多套路径。
- 目标：文本类请求统一走 chat completions 请求模型。

### 3.3 酒馆预设缺少后处理模式
- 现状：只有原始顺序注入和少量内置占位符处理。
- 目标：统一提供酒馆提示词后处理策略，至少包含：
  - 未选择
  - 单一用户
  - 严格
  - 半严格

## 4. 重构目标

### 4.1 保持不变
- 功能效果不变。
- 状态命令语义不变。
- 现有 UI 和存档使用方式不变。
- 提示词内容不擅自改写含义。

### 4.2 必须改变
- 大文件解耦。
- 重复逻辑收口。
- 文本 AI 请求统一为 chat completions。
- COT 选择改为直接引用不同文件。
- 酒馆后处理形成独立策略层。
- 旧兼容分支明显收缩。
