# 剧情规划 / 女主规划 / 小说分解 / 动态世界联动重构计划

## 1. 文档目的

这份文档用于重新定义以下四套核心变量结构及其联动方式：

1. 原创剧情规划
2. 原创女主剧情规划
3. 同人剧情规划 / 同人女主剧情规划
4. 小说分解 / 动态世界

本轮计划直接按“重构”处理，不做旧结构兼容，不保留旧版字段心智，不继续沿用 ID 驱动方案，也不保留 `V2` 之类版本号命名。

目标不是让结构“看起来整齐”，而是让 AI 真正能据此判断：

- 现在该推进什么
- 为什么能推进
- 为什么不能推进
- 什么时候才能推进
- 推进后会沉淀什么
- 何时切章 / 切组
- 切章后旧内容如何归档，新内容如何重建

---

## 2. 本轮重构的硬性结论

### 2.1 必须保留

- `镜头规划`
- 章节沉淀到历史
- 小说分解与同人规划的组号对应
- 规划分析 API / 独立世界演变 API 的前后章滑动注入
- 明确的触发时间 / 触发条件 / 前置条件 / 阻断条件

### 2.2 必须删除或停止沿用

- `V2` 之类版本号命名
- 依赖 `ID` 才能定位的规划结构
- `本回合主推`
- `当前场面关系`
- `势力常识`
- `地理常识`
- 在小说分解中加入 `阶段定位`
- 无实际约束意义的强枚举类型

### 2.3 必须拆开

- 原创剧情规划 和 同人剧情规划，必须是两套独立结构
- 原创女主规划 和 同人女主规划，必须是两套独立结构
- 同人版规划结构必须单独放到一个文件夹里，避免和原创规划互相污染

### 2.4 不拆开的只有一个

- 小说分解结构只保留一套

原因：

- 小说分解本质上是在拆原著内容资产
- 当启用附加小说并进入同人模式时，本来就是基于这份分解资产做规划
- 因此不需要再额外做一套“同人版小说分解结构”
- 正确做法是：保留一套小说分解结构，但让它能为同人规划提供足够精确的时间、事件、前后关联、章节承接信息

## 3. 当前结构为什么无效

## 3.1 剧情规划的问题

当前规划更像“人工阅读摘要”，不像“AI 可执行计划”。

主要问题：

- 缺少明确的执行时间
- 缺少明确的触发条件
- 缺少明确的前置条件
- 缺少明确的阻断条件
- 缺少完成判定
- 缺少失败后的转移处理
- 缺少切章条件
- 缺少切章后的沉淀规则
- 缺少与小说分解组号的对应关系

结果是：

- AI 只能模糊地“觉得差不多可以推进”
- 但不能精确判断“现在还不能推进”
- 也不能知道“应该等到哪一天 / 哪个前置条件达成后再推进”

## 3.2 女主规划的问题

当前女主规划最大的问题不是信息少，而是推进粒度错了。

`本回合主推` 这种字段本身就是错位设计，因为：

- 一个回合通常不足以完整推进一个女主
- 女主推进本来就应该跟随剧情阶段推进
- 女主推进不应按“这一回合推谁”来组织，而应按“这一剧情阶段主推谁、次推谁、禁推谁”来组织

同时当前女主规划还存在：

- 结构过于附属，像剧情规划的补丁，而不是独立规划
- 缺少与主剧情的阶段联动
- 缺少与小说分解组的对应
- 缺少与原著分歧线的对应
- 缺少事件触发时间、触发条件、前置条件
- 缺少镜头规划与互动事件之间的对照关系

## 3.3 小说分解的问题

当前小说分解虽然已经能输出概括、角色、时间线，但对 AI 推进仍然不够用。

核心缺失：

- 没有把“关键事件”拆成可执行事件
- 没有明确事件的精确触发时间
- 没有明确事件的最早 / 最晚触发时间
- 没有明确事件前置条件
- 没有明确事件触发条件
- 没有明确事件阻断条件
- 没有明确该组结束后要把什么给下一组参考
- 没有明确第一章 / 第一组是“开局内容”，而不是“中途章节”

这会导致：

- AI 只能看到“这章发生了什么”
- 却不知道“什么时候才可以发生”
- 更不知道“为什么现在还不可以发生”

## 3.4 动态世界的问题

当前动态世界结构偏结果记录，不偏执行控制。

缺点：

- 记录了事件正在进行
- 但没有定义它何时可以开始
- 也没有定义它为什么会失败 / 延后 / 偏转
- 缺少对 NPC 后台行动的明确执行时间与条件门槛
- 缺少与章节分解、剧情规划、同人分歧线的直接映射

结果是：

- 世界演变像“自由发挥”
- 不是“按原著时间门槛和状态门槛逐步推进”

## 4. 新结构设计原则

## 4.1 第一原则：为 AI 执行而设计，不为人工观感而设计

所有未来项都必须回答下面六个问题：

1. 这件事是什么
2. 最早什么时候能发生
3. 还需要什么前置条件
4. 什么条件满足后立即可触发
5. 什么情况下不能触发或需要延后
6. 触发后会沉淀什么结果

凡是回答不了这六个问题的字段，都不算有效执行结构。

## 4.2 第二原则：数组优先

优先使用数组结构承载：

- 当前章任务
- 跨章延续事项
- 关键事件
- 分歧线
- 女主阶段推进
- 女主互动事件
- 镜头规划
- 世界待执行事件
- NPC 后台行动

原因：

- AI 处理数组更稳定
- 命令路径天然可写成 `[i]`
- 不需要为了“定位”硬造 ID
- 更容易做增删改与排序

## 4.3 第三原则：不依赖 ID

新结构不再把 `节点ID / 事件ID / 镜头ID` 当成核心定位方式。

定位原则改为：

- 数组索引定位
- 标题 / 名称辅助识别
- 同人模式下再辅以 `关联分解组`

原因：

- AI 最容易编造、重复、遗漏的就是 ID
- 命令系统本来就支持数组路径
- 标题 + 索引 + 组号已经足够定位

## 4.4 第四原则：能不用枚举就不用枚举

以下字段统一改为 `string`：

- 事件类型
- 镜头类型
- 推进状态
- 分歧线状态
- 关系阶段
- NPC 当前阶段
- 世界事件类型

原因：

- 这类字段在实际使用中持续扩展
- 强枚举只会让提示词和结构一起僵化
- AI 也更容易在被限制的枚举内胡乱套词

## 4.5 第五原则：区分“章节核心状态”和“规划”

章节本身的状态与规划不是同一个东西。

因此需要拆开：

- `剧情` 只负责当前章节、切章、历史沉淀等核心状态
- 原创模式下用 `剧情规划`
- 同人模式下用 `同人剧情规划`
- 原创模式下用 `女主剧情规划`
- 同人模式下用 `同人女主剧情规划`

这样可以避免把“章节状态”和“待执行规划池”混成一棵树。

## 4.6 第六原则：同人规划必须带原著锚点

同人规划里的每个关键任务、女主推进、世界事件，都必须能回答：

- 它承接的是哪一个分解组
- 它对应的是哪一个原著事件
- 它是延续原著、偏转原著，还是已经形成分歧线

## 7. 小说分解重构方案

## 7.1 改什么

重写 `models/novelDecomposition.ts` 的“分段 / 登场角色 / 分组整体时间线 / 关键事件”结构，让小说分解结果不再只是摘要，而是能给规划分析与世界演变直接提供执行依据。

## 7.2 为什么当前设计不够

当前结构中最致命的问题不是“内容少”，而是“事件不是执行结构”。

例如现在能告诉 AI：

- 这一组发生了什么
- 有哪些角色
- 大概什么时间

但不能告诉 AI：

- 这件事最早何时允许发生
- 当前时间是否已到
- 没满足哪条前置条件
- 事件被什么阻断
- 这一组结束后要给下一组什么参考

## 7.3 新结构

```ts
export interface 小说分解数据集结构 {
  数据集标题: string;
  作品名: string;
  总章节数: number;
  分解组列表: 小说分解组结构[];
  全书概括: string[];
  核心角色: string[];
}

export interface 小说分解组结构 {
  组号: number;

  本组结束状态: string[];
  给下一组参考: string[];
  登场角色: string[];
  时间线起点: string;
  时间线终点: string;
  关键事件: 小说分解事件结构[];
  角色推进: 小说分解角色推进结构[];
  势力推进: 小说分解势力推进结构[];
  镜头规划: 小说分解镜头结构[];
}

export interface 小说分解事件结构 {
  事件名: string;
  事件说明: string;
  触发时间: string;
  最早触发时间: string;
  最晚触发时间: string;

  涉及势力: string[];
  涉及地点: string[];

  对下一组影响: string[];
}

export interface 小说分解角色推进结构 {
  角色名: string;
  本组前状态: string[];
  本组变化: string[];
```

补充说明：

- `登场角色` 必须是 AI 直接从当前组原文里提取的正式字段，不允许前端本地从原文启发式猜测。
- 这里的 `时间线起点 / 时间线终点` 指的是“当前分组章节整体的开始时间 / 结束时间”，不是事件列表摘要。
- 上一组的 `时间线终点` 必须直接传给下一组任务，作为下一组 `时间线起点` 的参考基准。

## 7.4 为什么这样改

原因分四点：

1. `组号` 成为同人模式下所有规划的统一锚点
2. `关键事件` 被真正拆成“可执行事件”
3. `给下一组参考` 用于明确前后组承接
4. `镜头规划` 保留氛围与场外动态，不再被视为可删内容

## 7.5 第一章 / 第一组的特殊逻辑

第一组必须明确视为开局组。

规则：

- `是否开局组 = true`
- 不生成“进入本组前置条件”
- 不生成“本章节开启条件”这种中途章节语义
- 重点生成 `开局已成立事实`

第一组的职责是告诉系统：

- 开局世界已是什么样
- 开局人物位置是什么
- 开局冲突已成立哪些
- 开局第一批可触发事件有哪些

## 7.6 第二组及后续组的逻辑

从第二组开始，必须显式带上前一组部分结果作为参考。

需要给后一组的参考至少包括：

- 前一组结束后已经成立的事实
- 前一组未结清的冲突
- 已经锁定但尚未触发的时间门槛
- 下一组必须承接的人物状态
- 下一组必须承接的地点状态
- 前一组的结束时间线

也就是说，后一组不是只看原文，而是要看：

- 前一组留下了什么
- 当前组文本写了什么
- 下一组大致会接什么

## 7.7 分解生成逻辑重写

### 第一组输入

```text
【上一组结束时间 / 本组时间线起点参考】
无

【当前章节原文】
...
```

输出重点：

- `开局已成立事实`
- `登场角色`
- `时间线起点`
- `时间线终点`
- `关键事件`
- `本组结束状态`
- `给下一组参考`

### 第二组及后续输入

```text
【前一组参考】
- 前一组结束状态
- 前一组给下一组参考
- 前一组关键事件中未结清项

【上一组结束时间 / 本组时间线起点参考】
...

【当前章节原文】
...

【下一章节标题参考】（可选）
...
```

输出重点：

- `前组延续事实`
- `登场角色`
- `时间线起点`
- `时间线终点`
- `关键事件`
- `本组结束状态`
- `给下一组参考`

### 运行约束

- 小说分解任务必须严格按组号顺序执行，不允许跳章、跳组。
- 一旦某一组处理失败，任务必须暂停或进入重试，不允许继续向后处理下一组。
- 重试时必须从失败组继续，而不是从后续组重新开始。
- 只有当前组成功写回后，后一组才允许进入处理。

## 7.8 必须删除的旧逻辑

当前 `services/novelDecompositionPipeline.ts` 中这类本地启发式拼装逻辑需要删掉或重写：

- `构建章节开启条件`
- `构建章节结束条件`

原因：

- 它们现在本质上是在把摘要句子拼成“看起来像条件”的文本
- 这不是可执行条件
- 会让后续 AI 错把“摘要”当“达成判定”

## 7.9 时间门槛判定规则

未来所有小说分解事件都按以下规则使用：

1. 当前时间早于 `最早触发时间`，则该事件不能触发
2. 当前时间达到 `最早触发时间` 但 `前置条件` 未满足，则该事件不能触发
3. `前置条件` 满足但 `触发条件` 未满足，则该事件处于待触发
4. 若命中任一 `阻断条件`，则该事件不能按原路径触发
5. 当前时间超过 `最晚触发时间` 且事件仍未发生，则该事件进入“错过 / 偏转”判断

这套规则后续必须直接给规划分析与世界演变复用。

## 8. 原创剧情规划重构方案

## 14. 切章 / 切组 / 历史沉淀规则

## 14.1 改什么

必须建立统一的“切章后沉淀 + 清空 + 重建”流程。

## 14.2 当前问题

现在最大的问题不是“不会切章”，而是：

- 切章后旧章内容没有系统沉淀
- 旧规划没有被清理
- 新章节规划没有被重建

这会直接导致：

- 新章还背着旧章待办
- AI 误以为旧章任务仍应继续执行
- 章节感丢失

## 14.3 新规则

当满足切章 / 换组条件时，必须按以下顺序执行：

1. 生成本章历史沉淀
2. 将本章已完成内容写入 `剧情.历史卷宗`
3. 将仍需跨章延续的内容提炼为 `跨章延续事项`
4. 清空本章专属规划
5. 切换到下一章 / 下一组
6. 基于下一章 / 下一组重建新的剧情规划
7. 基于下一章 / 下一组重建新的女主规划

## 14.4 本章历史沉淀内容必须包括

- 本章完成了什么
- 本章没有完成但延续了什么
- 本章人物关系发生了什么长期变化
- 本章势力 / 地点 / 资源发生了什么长期变化
- 若为同人模式，本章形成了哪些分歧线或分歧扩大

## 14.5 必须清空的内容

切章后以下内容应视为“本章专属规划”，需要清空并重建：

- 当前章任务
- 当前章待触发事件
- 当前章镜头规划
- 当前章女主互动事件
- 当前章女主镜头规划

## 14.6 允许保留的内容

只有以下内容允许跨章保留：

- 跨章延续事项
- 长期分歧线
- 跨章女主关系线
- 长期世界事件

```text
- 本组结束状态
- 给下一组参考
- 镜头规划
```

## 15.5 同人模式下额外包含

```text
- 当前对齐分解组
- 当前已形成分歧线
- 当前章允许偏转点
- 当前章禁止提前改写点
```

## 15.6 为什么这样改

规划分析不是看摘要，而是看“执行门槛”。

没有前后章滑窗时，AI 不知道：

- 当前章已经做到了哪里
- 下一章还没到哪里
- 哪些事只能预热，不能提前执行

---

## 16. 独立世界演变 API 注入改造

## 16.1 改什么

独立世界演变 API 同样必须改成滑动注入，固定使用：

- `【前一章节内容】`
- `【当前章节内容】`
- `【下一章节内容】`

## 16.2 世界演变侧的注入重点

和规划分析相比，世界演变更关注：

- 哪些世界事件已经到时间
- 哪些 NPC 行动可以开始
- 哪些世界变化只能作为预热，不能越级结算

因此当前章节内容至少要额外强调：

- 本组关键事件的时间门槛
- 本组人物 / 势力 / 地点结束状态
- 下一组即将需要的世界前置状态

## 16.3 这样改的原因

如果世界演变拿不到明确时间门槛和条件门槛，它就一定会出现两个问题：

1. 提前触发本不该发生的事件
2. 错过应当在后台推进的事件

---

## 17. 滑动注入的运行规则

滑动注入不只是“把三段文本拼一起”，而是一个章节定位机制。

运行规则如下：

1. 当前对齐组号 = 运行时记录的当前分解组
2. 注入时自动取 `前一组 / 当前组 / 下一组`
3. 若当前组已完成且换组条件满足，则下一轮注入平移到后一个窗口
4. 第一组无前一组
5. 最后一组无下一组

同人模式下，滑动窗口还要驱动：

- 同人剧情规划重建
- 同人女主规划重建
- 世界待执行事件刷新

## 20. 验证场景

## 场景 1：第一章 / 第一组分解

要求：

- 明确标记为开局组
- 没有“进入本组前置条件”式中途语义
- 只输出开局已成立事实与第一批可触发事件

## 场景 2：事件时间门槛

设定：

- 某事件 `最早触发时间 = 一年一月三日`

要求：

- 在 `一年一月一日` 不能触发
- 即使 AI“感觉气氛到了”也不能触发

## 场景 3：前置条件门槛

设定：

- 某事件要求先获得某线索

要求：

- 时间已到，但线索未获得时，事件不能触发

## 场景 4：同人分歧线

设定：

- 某原著事件本该发生，但本存档已经提前改写

要求：

- 同人剧情规划中出现明确 `分歧线`
- 世界事件与女主事件能引用该分歧线

## 场景 5：切章沉淀

要求：

- 当前章完成后，旧章内容写入 `剧情.历史卷宗`
- 当前章任务与镜头规划被清空
- 下一章规划被重新生成

## 场景 6：滑动注入

要求：

- 规划分析与世界演变都拿到 `前一章节内容 / 当前章节内容 / 下一章节内容`
- 切组后注入窗口自动平移

## 21. 本计划落地后的直接收益

## 21.1 对 AI

- AI 不再凭感觉推进
- AI 能明确判断“未到时间”
- AI 能明确判断“前置未满足”
- AI 能明确判断“该事件已错过或转入偏转”

## 21.2 对同人模式

- 原著锚点终于可追踪
- 分歧线终于可维护
- 女主推进终于能受原著关系线约束

## 21.3 对章节感

- 章节切换更干净
- 旧章会沉淀
- 新章会重建
- 不再出现多章规划混成一锅的情况

## 21.4 对动态世界

- 世界事件终于有执行门槛
- NPC 后台行动终于有时序
- 世界演变不再是自由漂移

## 22. 结论

本轮不是“补几个字段”，而是要把整条链路从“摘要式结构”改成“执行式结构”。

最终必须形成下面这套稳定关系：

1. 小说分解给出组号、事件、时间、条件、承接
2. 同人规划按分解组制定当前章任务与分歧线
3. 同人女主规划按分解组和分歧线制定阶段推进
4. 动态世界按同一套时间与条件推进 NPC 和世界事件
5. 规划分析与世界演变都使用 `前一章节内容 / 当前章节内容 / 下一章节内容` 滑动注入
6. 一旦切章，旧章沉淀，当前章清空，新章重建

只有这样，AI 才真正知道“如何推动 NPC 和世界事件的走向”，而不是继续依赖模糊概括去猜。
